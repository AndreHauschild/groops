GNSS satellite orbit determination and station network analysis

This cookbook chapter describes an example of global GNSS processing as done by analysis centers of the International GNSS Service (IGS). A scientific article detailing the underlying processing approach can be found via DOI 10.1007/s00190-018-1223-2.

Resulting products usually comprise:

  • Satellite orbits, clocks, and signal biases
  • Station positions, clocks, signal biases, and troposphere estimates
  • Earth orientation parameters
  • Normal equations

This example is limited to GPS and Galileo, a small station network of around 100 stations, and a single day at a 5-minute processing sampling.

Note: Global GNSS processing can become very computationally intensive. Depending on the number of satellites and stations, the observation and processing sampling, and certain parametrizations it can quickly exceed the capabilities of a normal desktop computer and may require computer clusters or number crunchers (see section Parallelization).

Data and metadata preparation

Global GNSS processing requires a large set of data and metadata. Most metadata and some data files are provided in GROOPS file formats at ftp://ftp.tugraz.at/outgoing/ITSG/groops/data (marked with * below). These files are regularly updated. Alternatively, check the linked file types and programs for information on how to create these files manually.

Satellite metadata:

  • Transmitter info*: PRN-SVN mapping, antenna offsets and orientations
  • Antenna definition*: antenna center variations
  • Receiver definition*: transmitted signal types (recommended)
  • Satellite macro model*: box-wing model, mass, antenna thrust
  • Attitude info*: attitude modes and parameters

Station metadata:

  • Station info*: antenna and receiver information, antenna reference point offsets, antenna orientations
  • Antenna definition*: antenna center offsets and variations
  • Receiver definition: observed signal types (optional)
  • Accuracy definition*: elevation and azimuth dependent accuracy

Most data files are not provided in GROOPS formats and have to be converted manually from external file formats.

Satellite data:

  • Approximate orbits: broadcast or precise IGS orbits in CRF for orbit integration (see Sp3Format2Orbit)
  • Approximate clocks: broadcast or precise IGS clocks (see GnssClockRinex2InstrumentClock)
  • Attitude: rotation from body frame to CRF (see SimulateStarCameraGnss or GnssOrbex2StarCamera)

Station data:

  • Precise position*: i.e. IGS reference frame position, for no-net constraints (see Sinex2StationPosition, Sinex2StationDiscontinuities, Sinex2StationPostSeismicDeformation)
  • Observations: receiver measurements (see RinexObservation2GnssReceiver)
  • Troposphere model*: mapping functions, i.e. VMF3 (see ViennaMappingFunctionGrid2File, ViennaMappingFunctionStation2File)

Additional data:

  • Earth orientation parameters*: IERS Earth rotation data (see EarthRotation)
  • Gravity field*: gravity field model for orbit integration (see Gravityfield)
  • Tides and tidal deformation*: models for astronomical, Earth, ocean, pole, ocean pole, and atmospheric tides (see Tides)
  • Albedo*: reflectivity and emissivity grids, i.e. from CERES (see Albedo)

Preprocessing: Orbit integration

Numerical integration of the satellite orbits is the first step in global GNSS processing. Use PreprocessingVariationalEquation and provide all relevant forces, the inputfileSatelliteModel, attitude (see SimulateStarCameraGnss), and approximate orbit. Setting the largest force components (i.e. the static gravity field) in gradientField is sufficient for GNSS satellites.

Force models usually include:

  • gravityField:potentialCoefficients: static gravity field (e.g. GOCO06s)
  • gravityField:trend
    • gravityField:potentialCoefficients: trend component of time-variable gravity field (e.g. GOCO06s)
  • gravityField:oscillation
    • gravityFieldCos:potentialCoefficients: annual cosine component of time-variable gravity field (e.g. GOCO06s)
    • gravityFieldSin:potentialCoefficients: annual sine component of time-variable gravity field (e.g. GOCO06s)
  • tides:astronomicalTide: astronomical tides (e.g. based on JPL ephemeris)
  • tides:earthTide: Earth tide (IERS conventions)
  • tides:doodsonHarmonicTide: ocean tides (e.g. FES 2014b)
  • tides:doodsonHarmonicTide: atmospheric tides (e.g. AOD1B RL06)
  • tides:poleTide: pole tides (IERS conventions)
  • tides:poleOceanTide: ocean pole tides (IERS conventions)
  • miscAccelerations:solarRadiationPressure: solar radiation pressure (box-wing model)
  • miscAccelerations:albedo: Earth radiation pressure (albedo model)
  • miscAccelerations:antennaThrust: antenna thrust (e.g. from IGS metadata SINEX file)
  • miscAccelerations:relativisticEffect: relativistic effects (IERS conventions)

The resulting variational equations file contains the integrated orbit, derivatives with respect to the satellite state vector, attitude, Earth rotation, and satellite model. Use PreprocessingVariationalEquationOrbitFit to fit the integrated orbit to the approximate orbit by least squares adjustment. This adjustment usually includes additional empirical parameters for solar radiation pressure to improve the orbit fit. Add parametrizationAcceleration:gnssSolarRadiation and select the ECOM2 parameters to be estimated. The resulting variational equations file contains the updated orbit and derivatives.

It is recommended to perform the steps above in a loop over all satellites/PRNs using LoopPrograms. Since PreprocessingVariationalEquation requires an inputfileSatelliteModel and SimulateStarCameraGnss requires an inputfileAttitudeInfo, both of which are generally defined for a specific satellite/SVN or block, GnssPrn2SvnBlockVariables can be used before these programs to define variables for SVN and block based on the PRN. These variables are then available in inputfileSatelliteModel and inputfileAttitudeInfo.

GNSS processing

Once all metadata and data are prepared and the orbits are integrated, the actual GNSS processing can begin. Use GnssProcessing and define the transmitter constellations, a receiver:stationNetwork, and the list of processingSteps.

Since this program is very complex, an overview of the most important settings used for the example processing in this cookbook chapter follows. Only config elements that deviate from the default values are listed. For most input files the default value and annotation give a good hint of what is expected, so they are not listed here.

The first step is setting the processing sampling, in this example it is 5 minutes. Alternatively, if observation preprocessing (outlier detection, cycle slip detection etc.) should be performed at full (i.e. 30-second) sampling, set sampling=30/86400 and define a processingStep:selectEpochs which selects every 10th epoch. The processing interval usually is a single 24-hour day, timeSeries:uniformSampling with timeStart=<mjd>, timeEnd=<mjd>+1, sampling=5/1440 (processing sampling).

Next define the GNSS constellations, in this case GPS and Galileo.

  • transmitter:GPS
    • inputfileSignalBias: leave empty, since we estimate our own biases
    • inputfileVariational=preprocessing/variational.{prn}.dat (integrated orbits)
    • estimateOrbit:dynamic
      • parametrizationAcceleration:gnssSolarRadiation: select same parameters used in orbit integration
      • stochasticPulse: set epoch for stochastic pulse (instantaneous velocity change) parameters to further improve orbit fit (usually at 12:00)
    • estimateCodeBias=yes
    • estimatePhaseBias=yes
    • biasModel: time-variable GPS L5 phase bias
      • type=L5*G
      • parametrizationTemporal:splines (e.g. degree 3 with hourly nodes and daily intervals)
    • estimateClockError:epochWise
      • sigmaZeroMeanConstraint=0.0001 (zero-mean constraint over GPS satellite clocks to remove rank deficiency in clock parameters)
    • antennaCenter (optional, to add antenna centers to normal equations)
      • antennaCenterVariations:center (enable X, Y, Z)
      • patternTypes=L1*G, L2*G, L5*G
      • addNonMatchingTypes=no
  • transmitter:Galileo (use same settings as for GPS, except for what is listed below)
    • biasModel is removed (not needed for Galileo)
    • estimateClockError:epochWise
      • sigmaZeroMeanConstraint=0 (no constraint needed, clock rank deficiency is solved by GPS zero-mean constraint)

Define the station network to be processed. In this example it will contain around 100 stations. A subset of them (i.e. those that have IGS reference frame coordinates) are going to be used for no-net constraints. Prepare the respective station list files (one station name per line) beforehand.

  • receiver:stationNetwork
    • inputfileStationList: list of all stations to be processed
    • inputfileSignalBias: leave empty, since we estimate our own biases
    • positionEstimation:constant
    • noNetTranslationSigma=0.00001 (optional)
    • noNetRotationSigma=0.00001 (required to solve rank deficiency)
    • noNetStationList: list of no-net stations (e.g. IGS reference frame stations)
    • estimateCodeBias=yes
    • estimatePhaseBias=yes
    • troposphere:viennaMapping (e.g. VMF3 operational gridded)
    • troposphereWetEstimation:splines (e.g. degree 1 with 2-hourly nodes and daily intervals)
    • troposphereGradientEstimation:constant
    • troposphereGradientEstimation:trend (at center of day with timeStep=1)
    • tides:earthTide: Earth tidal deformations (IERS conventions)
    • tides:doodsonHarmonicTide: ocean tidal deformations (e.g. FES 2014b, minDegree=1)
    • tides:doodsonHarmonicTide: atmospheric tidal deformation (e.g. AOD1B RL06, minDegree=1)
    • tides:poleTide: pole tidal deformations (IERS conventions)
    • tides:poleOceanTide: ocean pole tidal deformations (IERS conventions)
    • preprocessing: settings for station-wise observation preprocessing (e.g. ignoreType=*6*E to ignore Galileo E6 observations)

In the next step, define the Earth rotation and all Earth orientation parameters (EOPs) to be estimated. For GNSS this generally includes polar motion, polar motion rate, and length of day (LOD). A tightly-constrained parameter for dUT1 is added as well so it is part of the resulting normal equations. The selected Earth rotation includes a high-frequency EOP model.

  • earthRotation
    • earthRotation:file with inputfileEOP=timeSeries_EOP_IAU2000_desai.dat
    • earthRotationModels:itrf2010b with empty inputfileEOP (model will be reduced from output for a priori EOP parameters)
    • estimatePole:constant: polar motion
    • estimatePole:trend polar motion rate (at center of day with timeStep=1)
    • estimateUT1:constant: dUT1 (not estimable with GNSS, only for normal equations)
    • estimateUT1:trend: length of day (at center of day with timeStep=-1 to match IGS sign convention)

Some parametrizations don't offer config elements for constraints on the respective parameters, so they have to be added as general constraints. The parameters to be constrained can be selected by name (including support for wildcards). The sigma for general constraints is always of the same unit as the parameter.

  • constraints: loose constraint on GPS L5 phase biases
    • parameters:wildcard with object=G??, type=signalBias.L5*, temporal=*, interval=*
    • sigma=5 meters
    • relativeToApriori=yes
  • constraints: loose constraint troposphere estimates
    • parameters:wildcard with object=*, type=troposphere*, temporal=*, interval=*
    • sigma=5 meters (wet)/degrees (gradient)
    • relativeToApriori=yes
  • constraints: constraint on stochastic pulses
    • parameters:wildcard with object=*, type=stochasticPulse*, temporal=*, interval=*
    • sigma=0.1 micrometers/second
  • constraints: tight constraint on dUT1 parameter (not estimable with GNSS, only for normal equations)
    • parameters:wildcard with object=earth, type=UT1, temporal=, interval=*
    • sigma=1e-5 milliseconds
    • relativeToApriori=yes
  • constraints: tight constraint on antenna center (only for normal equations)
    • parameters:wildcard with object=*, type=antennaCenter*, temporal=*, interval=*
    • sigma=1e-6 meters

Finally, define the processing steps. This can be overwhelming at first, but offers a lot of flexibility. The general processing strategy is to first process a well-distributed subset of stations (i.e. a core network) to get good estimates of all satellite parameters, which then enables integer ambiguity resolution (IAR). Once the ambiguities of the core network are resolved and there are stable estimates for satellite phase biases, all other (non-core) stations can be processed individually (including IAR) while keeping the satellite parameters fixed. Finally, all stations can be processed together with all satellite parameters.

Let's start with the core network.

  • processingStep:selectNormalsBlockStructure: block structure for normal equations (different values may be more computationally efficient depending on the platform)
    • defaultBlockSizeInterval=512
    • defaultBlockSizeAmbiguity=256
    • keepEpochNormalsinMemory=no (in case your platform has limited memory, otherwise yes)
  • processingStep:selectParameters: parameters for least squares adjustment (TEC biases cannot be estimated together with signal biases)
    • estimateReceiverTecBias=no
    • estimateReceiverAntennaCenterVariations=no
    • estimateTransmitterTecBias=no
    • estimateTransmitterAntennaCenterVariations=no
    • constraintIonosphereSTEC=no
    • estimateIonosphereVTec=no
    • estimateIonosphereTecMap=no
  • processingStep:selectReceivers: start with a core network of well distributed stations (50-60 stations covering all signal types recommended)
    • inputfileStationList: list of core network stations (one line per station with alternative stations on the same line separated by whitespace)
  • processingStep:estimate: initial iterative estimation of all parameters (except non-core station parameters, TEC biases and antenna centers)
    • computeResiduals=yes
    • adjustSigma0=groupedPhases
    • computeWeights=groupedPhases (downweight code observations individually, but phase observations all grouped together)
    • convergenceThreshold=0.005 meters
    • maxIterationCount=8
  • processingStep:group: estimate TEC biases in an intermediate step (set this as a global element as we will use it again further below)
    • processingStep:selectParameters: set everything to no except for the elements below (settings apply only to steps in current group)
      • estimateReceiverTecBias=yes
      • constraintTransmitterSignalBiasZeroMean=yes
      • estimateTransmitterTecBias=yes
      • estimateIonosphereVTec=yes
      • estimateIonosphereTecMap=yes
      • constraintOther=yes
    • processingStep:estimate
      • computeResiduals=yes
      • convergenceThreshold=0.005 meters
      • maxIterationCount=3
  • processingStep:estimate: some more core network iterations (same settings as first time except for maxIterationCount=2)
  • processingStep:resolveAmbiguities: integer ambiguity resolution with core network
    • computeResiduals=yes
    • adjustSigma0=groupedPhases
    • computeWeights=groupedPhases
  • processingStep:estimate: some more core network iterations (same settings as first time except for maxIterationCount=2)

Now all other (non-core) stations can be processed separately.

  • processingStep:forEachReceiverSeparately: process all non-core stations individually with fixed transmitter parameters
    • inputfileExcludeStationList: list of core network stations (same file as inputfileStationList in processingStep:selectReceivers)
    • Use the same steps here as for the core network, except for selectParameters (settings are still valid here) and processingStep:selectReceivers (set automatically by forEachReceiverSeparately). One option is to set the steps from the core network as global elements and simply link them here.

Next all stations are processed together.

  • processingStep:selectReceivers with empty inputfileStationList and inputfileExcludeStationList (selects all stations)
  • processingStep:estimate: some iterations with the full network
    • computeResiduals=yes
    • adjustSigma0=groupedPhases
    • computeWeights=groupedPhases
    • convergenceThreshold=0.005 meters
    • maxIterationCount=2
  • processingStep:group: estimate TEC biases again (link to the group you set as a global element in the core network steps here so you don't have to define it again)
  • processingStep:estimate: some final iterations, same settings as above

With all processing done, print the residual statistics and write the resulting output files.

  • processingStep:printResidualStatistics
  • processingStep:writeResults

Finally, since we also want to provide normal equations, some additional steps are necessary. Our normal equations shall include all parameters except for clocks, (remaining float) ambiguities, and receiver signal biases due to size restraints. These parameters will be removed via parameter elimination.

  • processingStep:group: this group is optional, but this way the parameters selected below only affect the normal equations
    • processingStep:selectParameters: antenna center offsets are now included since we want them in the normal equations
      • estimateReceiverTecBias=no
      • estimateTransmitterTecBias=no
      • constraintIonosphereSTEC=no
      • estimateIonosphereVTec=no
      • estimateIonosphereTecMap=no
    • processingStep:writeNormalEquations
      • eliminateEpochParameters=yes (clocks)
      • remainingParameters:complement: keep all parameters except for these
        • parameters:wildcard with object=*, type=ambiguity*, temporal=*, interval=* (ambiguities)
        • parameters:wildcard with object=????, type=signalBias*, temporal=*, interval=* (receiver signal biases)
