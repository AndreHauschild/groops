GnssParametrizationTransmitter

Definition and parametrization of GNSS satelllite constellations. For precise point positioning (PPP) the constellations are held fixed and no additional parameters are set up. In case of GNSS satellite orbit determination orbits, clock errors, and signal biases are usually estimated.

A list of satellite PRNs (i.e for GPS: G01, G02, G03, ...) must be provided via inputfileTransmitterList. Satellite system codes follow the RINEX 3 definition, see GnssType. All input and output files except inputfileAntennaDefinition, inputfileReceiverDefinition, and outputfileUsedTransmitterList are read/written for each satellite. The file name is interpreted as a template with the variable {prn} being replaced by the satellite PRN.

In case of PPP precise orbits must be provided via inputfileVariational. Otherwise, dynamic orbits can be estimated by setting estimateOrbit:dynamic, with inputfileVariational coming from orbit integration. The parametrizationAcceleration must include at least those parameters that were estimated in PreprocessingVariationalEquationOrbitFit. Additional stochasticPulse parameters can be set up to reduce orbit mismodeling.

Precise satellite clock errors have to be provided via inputfileClock in case of PPP. Otherwise, they can be estimated via estimateClockError, either epochWise or based on a noiseModel. In this case it is enough to provide approximate clock errors (e.g. from broadcast ephemeris) via inputfileClock. The clock datum can be defined via a zero-mean constraint over all satellites of a constellation using sigmaZeroMeanConstraint. This constraint is applied per epoch.

Signal biases for code and phase observations have to be provided via inputfileSignalBias in case of PPP. Otherwise they can be estimated by setting estimateCodeBias=yes and estimatePhaseBias=yes, in which case no a priori signal biases are needed.

Antenna center offsets and variations can be estimated by defining antennaCenter. It is possible to estimate patterns for each individual satellite or grouped by satellite block. Since it is not possible to estimate, for example, antenna center offets based on a single day, they are usually hard-constrained to their a priori value. This can be done by adding constraints in GnssProcessing.

Input files used in all systems:

  • inputfileTransmitterInfo: Created via GnssAntex2AntennaDefinition or GnssStationInfoCreate.
  • inputfileAntennaDefinition: Created via GnssAntex2AntennaDefinition or GnssAntennaDefinitionCreate.
  • inputfileReceiverDefinition: Created via GnssReceiverDefinitionCreate in case you want to define which signal types a satellite transmits.
  • inputfileSignalBias: Converted via (TODO: SinexBias2GnssSignalBias) or estimated via GnssProcessing.
  • inputfileVariational: Integrated via PreprocessingVariationalEquation and PreprocessingVariationalEquationOrbitFit, created via (TODO: GnssVariationalCreate), or output of GnssProcessing.
  • inputfileClock: Converted via GnssClockRinex2InstrumentClock or GnssRinexNavigation2OrbitClock or output of GnssProcessing.

See also GnssProcessing and GnssSimulateReceiver.

GPS

Since GPS block IIF satellites are affected by time-variable signal biases (see Montenbruck et al. 2011, DOI: 10.1007/s10291-011-0232-x), a biasModel can be set up for phase observations on the L5 frequency by defining type=L5*G and an appropriate parametrizationAcceleration. The resulting outputfileBias contains the time-variable part of the respective bias evaluated at each epoch, while the constant part is included in outputfileSignalBias. In case of PPP the time-variable part can then be provided via timeVariableBias.
